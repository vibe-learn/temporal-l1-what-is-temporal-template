        # temporal — Что такое Temporal

        Homework-шаблон для урока **l1_what_is_temporal** (Что такое Temporal) на платформе Vibe Learn.

        ## Что делать

        Реализуй свой первый durable workflow на Go SDK (go.temporal.io/sdk). OrderWorkflow
оркеструет три активности (ChargeCard, ReserveStock, SendEmail), каждая — заглушка с
логом и искусственной флакой (падает с вероятностью p). Запусти локальный Temporal
(docker-compose в шаблоне поднимает temporalite/temporal-server + UI). Тесты проверят:
workflow доходит до конца несмотря на флаки активностей; при падении воркера
(симулируется в тесте через TestWorkflowEnvironment) ChargeCard не вызывается дважды.

## Контекст (из transfer-задачи урока)

У тебя сервис подписок. Процесс продления: списать карту → продлить доступ в БД →
отправить чек → если списание не прошло за 3 попытки в течение суток, понизить тариф
до free и уведомить. Сейчас это реализовано так: cron раз в час сканит таблицу
`subscriptions` со статусами (`pending_charge`, `charging`, `charged`, `failed`),
отдельный воркер дёргает платёжку, ретраи и дедлайны размазаны по коду, а недавно
был инцидент: под перезапустился во время `charging`, и часть юзеров списались дважды.

**Вопрос:** как ты переложишь этот процесс на Temporal? Опиши:
(a) что станет workflow, а что — активностями;
(b) как Temporal убирает двойное списание после рестарта пода;
(c) как выразить «3 попытки в течение суток, потом downgrade» средствами Temporal
    (а не самодельным cron + счётчик в БД).

## Recap из урока

- Temporal — это **durable execution**: пишешь процесс как обычный последовательный код, движок доводит его до конца через падения, деплои и сбои сети.
- Состояние хранится как **event history** в Temporal Server (поверх БД), а не в твоих переменных. Падение воркера → новый воркер делает **replay** истории и продолжает с места остановки.
- Replay подставляет уже записанные результаты активностей → **побочные эффекты не повторяются** (нет двойного списания), без самодельных idempotency-стейтмашин.
- Четыре актора: **Server** (оркестрация+персист), **Worker** (твой процесс), **Workflow** (детерминированный дирижёр), **Activity** (побочные эффекты с ретраями).
- Temporal — НЕ очередь, НЕ cron, НЕ БД. Он оркеструет многошаговые/долгоживущие процессы поверх твоих сервисов. Для stateless запрос-ответ — оверкилл.

        ## Как работать

        1. Платформа Vibe Learn создаёт копию этого репо в твоём GitHub-аккаунте по клику «Начать домашку» на странице урока (через GitHub `/generate`, codecrafters-pattern).
        2. Склонируй копию локально, реализуй TODO в `main.go` (workflow + активности), прогони тесты, запушь.
        3. CI (`.github/workflows/ci.yml`) запускает `go vet` + `go test ./...` на каждый push. Платформа слушает результат через webhook от GitHub Actions и обновляет статус домашки на странице урока.

        ## Локальное окружение

        - Go 1.22+
        - SDK: `go.temporal.io/sdk`
        - Docker + docker-compose — `docker compose up` поднимает Temporal dev server на `:7233` + Web UI на `:8233`. Адрес переопределяется через env `TEMPORAL_ADDRESS` (дефолт `localhost:7233`).
        - Юнит-тесты на `testsuite.TestWorkflowEnvironment` (активности замоканы) бегут в CI БЕЗ сервера; интеграционный тест включается через `TEMPORAL_INTEGRATION=1`.

        ## Запуск

        ```bash
        # Поднять локальный Temporal dev server + UI
        docker compose up -d
        # Web UI: http://localhost:8233

        # Прогнать тесты (юнит на TestWorkflowEnvironment — без сервера;
        # интеграционный включается через TEMPORAL_INTEGRATION=1)
        go test ./...
        TEMPORAL_INTEGRATION=1 go test ./...

        # Запустить воркер (регистрирует workflow + активности, слушает task queue)
        go run .
        ```
